# Jamhawi i18n (Language System)

This skill documents the full translation system, how to use it in components, and when/how to run the scripts in `scripts/` to keep translations in sync.

**Always reference this skill before adding any new UI text to any component.**

---

## Overview

- Library: `i18next` + `react-i18next` + `i18next-browser-languagedetector`
- Config file: `src/i18n.ts`
- Languages: **Arabic (`ar`)** and **English (`en`)**
- Default / fallback language: **Arabic** (`fallbackLng: "ar"`)
- All keys live inline inside `src/i18n.ts` — there are no separate JSON locale files

---

## How to Use Translations in Components

### Basic usage
```tsx
import { useTranslation } from "react-i18next";

const { t, i18n } = useTranslation();

// Render a translated string
<h1>{t("products")}</h1>

// Check the current language
const isArabic = i18n.language === "ar";
```

### Bilingual data fields (Firestore documents)
Products and categories have both `name` (English) and `nameAr` (Arabic) fields. Always use this pattern:

```tsx
const displayName = i18n.language === "ar" ? product.nameAr || product.name : product.name;
```

The `|| product.name` fallback handles documents where `nameAr` was never filled in.

### Language toggle
The toggle lives in the navbar (`Layout.tsx`). It calls:
```ts
i18n.changeLanguage(i18n.language === "ar" ? "en" : "ar");
```
The button shows the **opposite** of the current language (`"عربي"` when English is active, `"EN"` when Arabic is active).

### Layout direction
**The layout direction is always `ltr` — never change it.** The `useEffect` in `Layout.tsx` sets `document.documentElement.dir = "ltr"` unconditionally. Switching language only swaps text; it never flips the layout.

## Key Naming Convention

Keys are `snake_case`. When a concept already has a key (e.g. `total`) and a new context needs a similar but different label, a `_2` suffix is used (`total_2`). This is an existing convention — follow it rather than trying to rename.

Keys are truncated to ~40 characters when auto-generated by the scripts. For example:
```
"Import a report or generate random data..."  →  import_a_report_or_generate_random_data_
"Impacts customer satisfaction directly..."   →  impacts_customer_satisfaction_directly_d
```

## Adding New Keys Manually

When you add UI text to a component, always:

1. Add the key to **both** `ar.translation` and `en.translation` in `src/i18n.ts`
2. Use `t("your_key")` in the component — never a hardcoded string

```ts
// src/i18n.ts — add to both blocks:
ar: {
  translation: {
    // ... existing keys ...
    pricing: "التسعير",
  }
},
en: {
  translation: {
    // ... existing keys ...
    pricing: "Pricing",
  }
},
```

```tsx
// In the component:
<h1>{t("pricing")}</h1>
```

## The `scripts/` Folder

Location: `E:\mangingproject\jamhawyy\react-example\scripts\`

These are Node.js maintenance scripts you run manually from the terminal. They help keep `i18n.ts` in sync when strings get added to source files without being translated.

| File | Purpose |
|---|---|
| `detect-missing-i18n.mjs` | Scans all `.tsx/.ts` under `src/` and finds hardcoded English strings not wrapped in `t()` |
| `add-missing-i18n.mjs` | Reads `missing-i18n-keys.json` and patches the detected keys into `src/i18n.ts` automatically |
| `missing-i18n-keys.json` | Intermediate report produced by the detect script; consumed by the add script |

---

## When to Run the Scripts

### Run `detect-missing-i18n.mjs` when:
- You just added a new page or component with a lot of UI text and want to audit what's missing
- You inherited or merged code from elsewhere and want to check coverage
- Before a release, to verify no strings were left hardcoded

```cmd
node scripts/detect-missing-i18n.mjs
```

Output:
- Console: grouped list of files and detected strings
- File: `scripts/missing-i18n-keys.json` with `en` keys filled, `ar` values blank

### Run `add-missing-i18n.mjs` when:
- The detect script found missing keys AND you have filled in the Arabic translations in `missing-i18n-keys.json` (or `add-missing-i18n.mjs` already has them in its `AR_TRANSLATIONS` map)
- You want the script to patch `i18n.ts` automatically instead of manually copy-pasting

```cmd
node scripts/add-missing-i18n.mjs
```

Output:
- `src/i18n.ts` is updated with new keys injected into both `ar` and `en` blocks
- `src/i18n.ts.bak` is created as a safety backup before any change

### Typical workflow for a new batch of strings:
```
1. node scripts/detect-missing-i18n.mjs
   → review console output + scripts/missing-i18n-keys.json

2. Fill in the "ar" values in missing-i18n-keys.json
   (or add them to the AR_TRANSLATIONS map in add-missing-i18n.mjs)

3. node scripts/add-missing-i18n.mjs
   → keys are injected into src/i18n.ts

4. Verify src/i18n.ts looks correct, delete the .bak if satisfied
```

---

## How the Scripts Work (internals)

### `detect-missing-i18n.mjs`
- Walks all `.tsx`/`.ts` files under `src/` recursively
- Skips lines with existing `t("...")` calls, imports, and comments
- Detects three pattern types:
  1. JSX text nodes: `>English Text<`
  2. String props: `label="English Text"` (excludes `className`, `src`, `href`, etc.)
  3. Inline JSX expressions: `{"English Text"}`
- Only flags strings that start with a capital letter and are ≥3 chars
- Skips strings already present as values in the `en` translation block
- Auto-generates `snake_case` keys (max 40 chars) with `_2`, `_3` suffixes for duplicates
- Writes the full report to `scripts/missing-i18n-keys.json`

### `add-missing-i18n.mjs`
- Reads `scripts/missing-i18n-keys.json`
- Has a hardcoded `AR_TRANSLATIONS` map with manually curated Arabic values
- Injects the new key-value pairs into both `ar.translation` and `en.translation` inside `src/i18n.ts` using string manipulation (finds the closing braces of each block)
- Falls back to the English value if no Arabic translation is provided in the map
- Always writes a `.bak` before modifying the file

## Rules

1. **Never hardcode English or Arabic text** in a component — always use `t("key")`
2. **Both languages must always have the key** — adding to `en` without adding to `ar` will cause the `ar` fallback to show the raw key name
3. **Layout direction is always LTR** — do not use language to flip `dir`
4. **Key names are permanent** — renaming a key breaks all existing usages; add a new key instead
5. **Run the detect script before asking Kiro to add i18n keys** — it gives the exact list of what's missing and where
6. **The scripts only cover `src/`** — they don't scan `scripts/` or `dist/`
7. **`missing-i18n-keys.json` is ephemeral** — it's overwritten on every detect run; don't store permanent data there
